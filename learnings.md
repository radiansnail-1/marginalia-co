# Learnings

- **AAB is not needed for assetlinks-only fixes** - The browser URL bar in a Play-installed TWA is controlled by Digital Asset Links verification between the website and the Play-signed Android app. Updating `public/.well-known/assetlinks.json` and redeploying the website is enough when package ID and wrapper config are unchanged.
- **Use Play App Signing SHA-256** - For Play installs, `assetlinks.json` must use the SHA-256 from Play Console's App signing key certificate, not the local upload key certificate. Current Play signing fingerprint is `8C:E6:E0:FF:94:A9:25:13:5D:5A:EF:E6:BA:D8:71:30:EA:71:A5:B6:A5:58:B1:16:A0:C6:E5:CE:FD:67:03:81`.
- **Package ID is fixed for this app** - The Play package and TWA verification package are `com.app.marginaliaandco`; old references to `com.radiansnail.marginalia` are stale.
- **Live verification requires deployment** - Local generation can pass, but TWA verification on-device will fail until `https://marginalia-co.vercel.app/.well-known/assetlinks.json` returns `200` JSON from Vercel.
- **API callers trust `/api/v1`** - If a field is supported by route code but missing from the machine-readable reference, LLM/tool callers treat it as unsupported. Keep `src/lib/api/reference.ts` in sync with `src/app/api/v1/books/route.ts`.
- **Book updates should preserve omitted fields** - POST `/api/v1/books` is used for safe updates from tools; omitted `status`, `rating`, and `review` should preserve existing values on updates instead of resetting them.
- **Half-star writes need DB migration 0009** - Route validation can accept `4.5`, but production still needs `supabase/migrations/0009_half_star_ratings.sql` applied so `user_books.rating` is `numeric(2,1)` with the half-star check constraint.
- **Onboarding is website-first for TWA** - Because the Play app is a Trusted Web Activity, `/onboarding`, `/invite/[code]`, referrals, rating suppression, and API contract changes are website changes that automatically affect the phone app after deploy. No AAB rebuild is needed unless native config, billing bridge, icons, or deep links change.
- **Migration order matters for onboarding** - `/onboarding` depends on `supabase/migrations/0015_onboarding_growth.sql`. The app layout and onboarding page currently degrade to `/home` if the migration is missing, and this exact production symptom was confirmed after PR #18 merged: new accounts skipped onboarding because `profiles.onboarding_completed_at`, `referral_codes`, and `referral_events` were absent in Supabase.
- **Onboarding completion should lead referral side effects** - The critical user-facing write is `profiles.onboarding_completed_at`; referral claim/qualification should happen after that and be best-effort so referral table/service failures cannot trap a user outside the app.
- **Invite routes should validate before cookies** - `/invite/[code]` should only persist valid referral codes. Invalid short/malformed codes should still land on sign-in but must not set `marginalia_referral_code`.
- **Rating is prompt suppression only** - A web/TWA app cannot verify that a Play Store rating happened. `I rated!` should only set `rating_prompt_claimed_at` to suppress future prompts; it must not unlock Plus, referral credit, or any reward.
- **Referral rewards need verifiable events** - Referral credit should be based on confirmed accounts, onboarding completion, or paid conversion. Reject self-referrals and duplicate referred users; never tie referral reward to ratings or fake installs.
- **QA the web path for TWA changes** - Because the Play app is a TWA, normal web QA on sign-in, invite, onboarding, and `assetlinks.json` is the right source of truth for these changes. Physical phone checks are useful for viewport fit, but no AAB rebuild is needed.
- **Browser Use can struggle with email inputs** - In this session Browser Use rendered the page but failed to type/fill the email input due to an automation bridge issue. Treat full signed-in onboarding QA as pending until it is done manually, with another browser surface, or after the migration is applied and a reusable session exists.
- **Browser QA can wedge below the app layer** - If every `node_repl`/Browser Use call fails before JavaScript runs with `failed to write kernel assets` or `Transport closed`, inspect Codex helper processes before debugging app code. In this session stale `node_repl.exe` and stdio `codex.exe app-server` pairs left the MCP transport wedged; a fresh Codex thread/app restart is the practical recovery.
- **Do not re-export types from server-action files** - In a `"use server"` action file, even a type-only re-export can become a runtime server-action bundle failure under Turbopack. Keep shared types in `src/lib/**` and import them directly into client components instead of through action modules.
- **Promo entitlement needs migration 0016** - `NLBisthebestlibrary` can be accepted and saved as a promo cookie in app code, but the configured Supabase project must have `supabase/migrations/0016_permanent_promo_code.sql` applied before `plus_unlocked_at`, `plus_access_source`, and `plus_promo_code` can be recorded.
- **No new visual assets needed for onboarding** - Existing `Owl`, `Letter`, fonts, theme tokens, and inline SVG check/lock/star/book icons are enough for the first shippable onboarding UI. A generated Librarian seal or Plus patron badge is optional later, not required now.
- **Do not expose or reuse pasted signing passwords** - Any upload key password pasted into chat should be treated as compromised; reset only before the first successful Play upload, then preserve the Play upload key.
